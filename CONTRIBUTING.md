# Contributing to copilot-mcp-soc-pack

Thanks for considering a contribution! This project bundles **free, low-friction threat-intel APIs** into a single container so SOC teams can deploy useful Microsoft Security Copilot context with one click. Contributions that keep that bar — free, fast, easy to deploy — are very welcome.

## Project policy

- **Free / freemium upstream APIs only.** No commercial-tier-only sources. Free keys are OK if a SOC analyst can self-serve the key in under 2 minutes.
- **No scraping.** Use official, documented APIs. Respect upstream rate limits and Terms of Service.
- **No raw leak-site content.** Ransomware tooling returns metadata (group, victim, country, disclosure date) — never the leaked files or their contents.
- **No secrets in code, logs, or error messages.** API keys come from environment variables / Container App secrets only.
- **Minimum dependencies.** This is an OSS supply-chain target; every new dependency in `pyproject.toml` must justify itself.

## Repository layout

```text
src/
  app.py                # FastAPI + FastMCP entrypoint
  common/               # Shared HTTP client, TTL cache
  tools/<service>.py    # 1 module = 1 upstream API (REST router + MCP tools)
deploy/main.bicep       # Container Apps + Log Analytics (source of truth)
sc-plugin/              # Security Copilot manifest + agent assets
mcp-client-config/      # VS Code / Claude Desktop sample config
tests/                  # pytest
scripts/smoke.ps1       # Operator smoke probe (post-deploy)
.github/workflows/      # build-push (GHCR), lint-test, bicep-build
```

## Adding a new tool (external API)

Drop one new file at `src/tools/<service>.py` following the 3-tier shape used by every existing module — see [`src/tools/kev.py`](./src/tools/kev.py) as the canonical example:

1. **Pure async helper functions.** All real work lives here. Tests target these directly.
2. **FastAPI `router`** with `@router.get(...)` decorators. This becomes the OpenAPI surface that Security Copilot calls.
3. **`register_mcp_tools(mcp: FastMCP)`** that re-exposes the same helpers via `@mcp.tool(...)`.

Then wire it up in [`src/app.py`](./src/app.py) (2 lines):

```python
from src.tools import <new>
app.include_router(<new>.router, dependencies=[Depends(_require_api_key)])
<new>.register_mcp_tools(mcp)
```

### Naming conventions

- REST path: `/<service>/<action>` — e.g. `/kev/lookup`, `/abusech/threatfox/search`.
- MCP tool name: `<service>_<action>` (snake_case) — e.g. `kev_lookup`, `threatfox_search`.
- Prefer **concrete names** that read well in a Security Copilot prompt over abstract verbs (`ransomware_live_victims_by_country` beats `query`).
- Environment variable for upstream API keys: `<SERVICE>_AUTH_KEY` or `<SERVICE>_API_KEY`.

### #ExamplePrompts

Every REST route's OpenAPI `description` should embed a `#ExamplePrompts` block — Security Copilot's planner uses these to pick the right skill. Three concrete, varied examples is the sweet spot. See any existing tool for the format.

### Security Copilot wiring

If your tool needs to be invokable from Security Copilot prompts, add the corresponding skill to [`sc-plugin/manifest.yaml`](./sc-plugin/manifest.yaml). The manifest uses the **Security Copilot native** schema (`Descriptor` + `SkillGroups`) with API-key auth — see existing skills for the pattern.

## Required quality bar

Before opening a PR:

```bash
ruff check .
pytest -q
```

Both must pass cleanly. Add at least one test for any new tool — mock upstream HTTP with `httpx.MockTransport` or `pytest-httpx`, never hit the real API in CI. If you add a new endpoint, [`tests/test_smoke.py`](./tests/test_smoke.py) will automatically assert it is API-key-gated.

For Bicep changes, **edit `deploy/main.bicep` only** — `deploy/azuredeploy.json` is regenerated by GitHub Actions and must not be hand-edited.

## Coding style

- Python 3.12+, full type hints, `from __future__ import annotations` at the top.
- All I/O is `async`. Use the shared `httpx.AsyncClient` from `src/common/http.py:get_client()`.
- Pydantic v2 models for request/response shapes.
- Cache upstream responses with `TTLCache` — TTL ≤ half of the upstream's documented rate limit.
- Errors: `HTTPException` only. 400 for bad input, 401 for missing upstream key, 429 for upstream throttling, 503 for upstream unreachable.
- Logging via stdlib `logging`. Never `print()`. Never log secrets.

## Commit & PR conventions

- **Conventional Commits** in English, e.g.:
  - `feat: add shodan internetdb classifier`
  - `fix(kev): handle missing dateAdded gracefully`
  - `docs: clarify Security Copilot registration steps`
  - `chore: regenerate azuredeploy.json`
- Small, focused PRs. Large changes → open a draft issue first.
- PR description should include working evidence (curl output, screenshot of the Security Copilot agent answering, or pytest output).
- The PR template will prompt you for the rest.

## Releasing (maintainers)

1. Land changes on `master` via PR.
2. CI (`build-push.yml`) publishes `ghcr.io/nobufumimurata/copilot-mcp-soc-pack:latest` and `:sha-xxxxxxx`.
3. Tag `v{major}.{minor}.{patch}` (annotated) and push — CI adds the matching semver tag to GHCR.
4. `gh release create vX.Y.Z` with notes summarising user-visible changes.

## Out of scope

- Azure SDK for Python — Container App secrets are sufficient.
- External datastores (Redis, PostgreSQL, …) — in-memory TTL cache is enough for v1.
- OAuth / MSAL inbound — API key auth is the only inbound mechanism for now.
- Anything that requires a paid commercial threat-intel licence.

## Code of conduct

Be kind. Assume good faith. Security work is stressful enough; the issue tracker doesn't need to add to it.
