# v0.8 — Migration to the official Microsoft Security Copilot agent manifest schema

Status: **planning / preview** — see [`sc-plugin/msschema/manifest.yaml`](../sc-plugin/msschema/manifest.yaml)
for a draft.

Reference: [Agent Manifest](https://learn.microsoft.com/en-us/copilot/security/developer/agent-manifest)
(`Descriptor` / `SkillGroups` / `AgentDefinitions`).

## Goal

Allow operators to register the SOC Pack as a **single one-shot YAML
upload** via Build → My agents → Upload YAML, with both the API plugin
and reference triage agents bound automatically — no manual UI tool
binding.

## Why this is not yet possible end-to-end

Three facts collide:

1. **`sc-plugin/manifest.yaml` is already MS-schema** (Descriptor +
   SkillGroups[Format: API]). This part is fine and is what powers the
   currently-working Custom plugin upload.
2. **`sc-plugin/agent.yaml` uses a legacy/community shape**
   (`agents[].plugins[].skills`). Uploading it surfaces the warning
   "No agent tools detected from YAML import" because the Agent Builder
   UI cannot map this shape onto its current data model.
3. **FastAPI's auto-generated `operationId` values are dirty**:

   ```text
   kev_lookup_endpoint_kev_lookup_get
   epss_score_endpoint_epss_score_get
   d3fend_defenses_for_attack_endpoint_d3fend_defenses_for_attack__attack_technique_id__get
   ```

   These become the API skill names that an MS-schema agent's
   `ChildSkills` list must reference. Any AGENT manifest written today
   would need these ugly names hard-coded.

## v0.8 work plan

### Step 1 — Clean up FastAPI operationIds (prerequisite)

Add an explicit `operation_id=` to every router decorator in
`src/tools/*.py` so that the OpenAPI `operationId` matches the MCP
tool name:

```python
@router.get(
    "/kev/lookup",
    operation_id="kev_lookup",      # NEW
    response_model=KevLookupResponse,
    summary="...",
)
async def kev_lookup_endpoint(...): ...
```

Tasks:

- [ ] Add `operation_id=` to all ~30 endpoints across `src/tools/`.
- [ ] Update `tests/test_app.py::test_openapi_schema_contains_tools`
      to assert clean operationIds.
- [ ] Re-deploy Container App; re-upload the existing Custom plugin
      so SC reflects the new skill names.
- [ ] Update `docs/promptbook.md` Tier 3 prompts to reference the new
      clean names (they happen to match what the legacy agent.yaml
      already uses, so most prompts will not need changes).

This is a behaviour-preserving refactor for everything except the
displayed skill names in the SC plugin browser.

### Step 2 — Promote the MS-schema manifest to GA

After Step 1 ships:

- [ ] Move `sc-plugin/msschema/manifest.yaml` → `sc-plugin/manifest.yaml`
      (replacing the current API-only manifest, which is a strict
      subset of the new file).
- [ ] Archive `sc-plugin/agent.yaml` and `sc-plugin/ai-plugin.json`
      under `sc-plugin/legacy/` with a README pointer.
- [ ] Document a single upload step in the main README:
      `Build → My agents → Upload YAML → manifest.yaml`.

### Step 3 — Update README + docs

- [ ] Replace the current dual-step (Custom plugin + manual agent
      assembly) instructions with the single Upload YAML flow.
- [ ] Note in `docs/security-copilot-registration.md` that v0.7 users
      can keep their existing Custom plugin registration; v0.8 only
      adds the new pre-built agents on top.

## Out of scope for v0.8

- KQL / LogicApp skill formats (Format: KQL / LogicApp). The Container
  App is HTTP-only and the SOC Pack does not own any KQL tables or
  Logic Apps.
- Interactive agents (`PromptSkill`, `Interfaces: InteractiveAgent`).
  The current agents are single-input pipelines; promoting them to
  interactive form factors is a separate UX decision.

## Open questions

- Does the SC Agent Builder UI accept multiple `AgentDefinitions` in
  one YAML upload, or does each agent need its own file? The official
  sample bundles one per file; the draft `msschema/manifest.yaml`
  bundles three. Validate this when Step 1 unblocks Step 2.
- Whether `Descriptor.Name` collisions with the existing
  `copilot_mcp_soc_pack` plugin require renaming on install. The
  preview uses `CopilotMcpSocPack.IocTriage-v08preview` to avoid
  the collision while both are uploadable side-by-side.
