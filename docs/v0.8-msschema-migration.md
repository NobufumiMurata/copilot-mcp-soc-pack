# v0.8 — Microsoft Security Copilot agent manifest (status & lessons learned)

Status: **shipped as reference only — not the recommended runtime path.**
The sample manifest at [`sc-plugin/msschema/manifest.yaml`](../sc-plugin/msschema/manifest.yaml)
uploads cleanly via **Build → My agents → Upload YAML** and the
resulting agents publish and run end-to-end.

Reference: [Agent Manifest](https://learn.microsoft.com/en-us/copilot/security/developer/agent-manifest)
(`Descriptor` / `SkillGroups` / `AgentDefinitions`).

## What works

- The MS-schema manifest is accepted by the SC import pipeline.
- Agent publish succeeds with no "Invalid references" errors.
- `ChildSkills` resolve against the companion Custom plugin
  (`copilot_mcp_soc_pack`) listed under `RequiredSkillsets`.
- The upstream OpenAPI spec served by the Container App is now a
  clean OpenAPI 3.0.1 document with sanitised `operationId` values
  (`kev_lookup`, `abuseipdb_check`, …), an `ApiKeyAuth` security
  scheme advertised first, and a `servers[]` block populated from
  `MCP_SOC_PACK_PUBLIC_BASE_URL`. All five prerequisites that
  originally blocked v0.8 (operationId hygiene, ApiKey scheme,
  servers block, 3.0.1 strict validation, AGENT-only skillset)
  shipped via PRs #33–#37 and are covered by tests.

## Why we are not promoting it as the recommended path

In the Security Copilot standalone portal as of April 2026:

- The planner LLM tends to call **one** of the declared `ChildSkills`
  per agent invocation and treat its output as the final answer.
  Multi-tool fan-out / reconciliation does not happen reliably even
  when the manifest `Instructions` explicitly demand it.
- `Instructions` are static — there is no runtime hook to inject
  extra guidance per invocation; the user has to put the entire
  orchestration prompt into the structured `Inputs` field.
- `ChildSkills` is a "may-call" list, not a "must-call" list.

In short: the framework parts are healthy, the runtime model is the
limiting factor.

## Recommended runtime path: SOC Pack as MCP server

The same Container App also exposes Streamable-HTTP MCP at `/mcp/`
(see PR #38). Pointing a modern agentic client at it yields
substantially better triage output from the same toolset because the
client runs on a Claude Sonnet / GPT-4-class model with proper
multi-tool planning and recursive reasoning.

Sample client configurations:

- VS Code Copilot Chat (Agent Mode):
  [`mcp-client-config/vscode.mcp.json`](../mcp-client-config/vscode.mcp.json)
- Claude Desktop:
  [`mcp-client-config/claude-desktop.json`](../mcp-client-config/claude-desktop.json)

## What the manifest is still useful for

1. As a **reference implementation** for anyone building an MS-schema
   agent against the SOC Pack — it shows the correct wiring of
   `RequiredSkillsets`, `ChildSkills`, `AgentDefinitions`, and
   `Triggers`.
2. As a **regression target** for upcoming Security Copilot platform
   improvements (model upgrade, recursive planning, dynamic
   instructions). When the planner gains real multi-tool reasoning
   the same manifest should suddenly work end-to-end.
3. As a **smoke test** for the upload pipeline in CI / demos.
